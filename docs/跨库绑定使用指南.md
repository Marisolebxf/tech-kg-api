# 跨库绑定（Entity Binding）使用指南

## 概述

跨库绑定功能实现人才库、论文库、专利库、机构库之间的实体对齐绑定。采用 **规则召回 + LLM 精排** 的两阶段策略：先用姓名/机构/字段相似度召回候选对，再用 GLM 大模型精确判断是否为同一实体。

绑定结果以**边（Edge）**的形式写入 TRS Graph，边的属性包含置信度、匹配方法、规则分数、LLM 分数、状态等。

---

## 前置条件

- **TRS Graph** 数据库运行中（默认端口 9669）
- **trs-graph-service** Java REST 服务运行中（默认端口 8090）
- Python 3.12+ 环境，安装好项目依赖

### 启动 trs-graph-service

```bash
cd your_work_space/trs-graph-service
bash start.sh          # 启动
bash start.sh status   # 查看状态
bash start.sh stop     # 停止
```

服务启动后可通过 http://localhost:8090/health 检查健康状态。

---

## 配置

通过 `.env` 文件或环境变量配置图数据库连接，默认使用 TRS Graph：

```env
# 图数据库类型：trs_graph（默认）或 neo4j
GRAPH_DB_BACKEND=trs_graph

# TRS Graph 配置（trs-graph-service 地址）
GRAPH_DB_URI=http://localhost:8090
# TRS Graph 图空间名称
GRAPH_DB_DATABASE=entity_binding_demo
GRAPH_DB_CONNECTION_TIMEOUT=30
```

参考 `.env.example` 查看完整配置项。TRS Graph 的 `GRAPH_DB_URI` 填写 trs-graph-service 的 IP 和端口即可。

## API 接口

所有绑定接口前缀：`/api/v1/binding`

### 1. 初始化测试数据

```bash
curl -X POST http://localhost:9002/api/v1/binding/init-data
```

执行内容：
1. 创建 3 种绑定边类型（`bind_talent_paper_author`、`bind_talent_patent_inventor`、`bind_org_org`）
2. 创建 9 个索引（姓名、机构、作者等字段索引）
3. 插入测试节点：5 个 talent + 6 个 cn_paper + 4 个 patent + 4 个 cn_organization

返回示例：

```json
{
  "edge_types_created": ["bind_talent_paper_author", "bind_talent_patent_inventor", "bind_org_org"],
  "indexes_created": ["idx_talent_name", ...],
  "nodes_inserted": {"talent": 5, "cn_paper": 6, "patent": 4, "cn_organization": 4},
  "message": "初始化完成: 创建边类型3个, 索引9个, 插入节点19个"
}
```

### 2. 执行绑定

```bash
# 执行全部绑定
curl -X POST http://localhost:9002/api/v1/binding/execute \
  -H "Content-Type: application/json" \
  -d '{"binding_type": "all"}'

# 只执行人才↔论文绑定
curl -X POST http://localhost:9002/api/v1/binding/execute \
  -H "Content-Type: application/json" \
  -d '{"binding_type": "talent_paper"}'

# 只执行人才↔专利绑定
curl -X POST http://localhost:9002/api/v1/binding/execute \
  -H "Content-Type: application/json" \
  -d '{"binding_type": "talent_patent"}'

# 只执行机构↔机构绑定
curl -X POST http://localhost:9002/api/v1/binding/execute \
  -H "Content-Type: application/json" \
  -d '{"binding_type": "org_org"}'
```

绑定 Pipeline 流程：

```
数据召回（从 TRS Graph 批量拉取节点）
  ↓
规则召回（姓名匹配 + 机构相似度 + 字段相似度）
  ↓
LLM 精排（调用 GLM 判断是否同一实体，未配置时 fallback 到规则评分）
  ↓
结果写入（创建绑定边，写入置信度和状态）
```

返回示例：

```json
{
  "talent_paper": {"binding_type": "talent_paper", "total_candidates": 6, "confirmed": 6, ...},
  "talent_patent": {"binding_type": "talent_patent", "total_candidates": 3, "confirmed": 3, ...},
  "org_org":       {"binding_type": "org_org", "total_candidates": 4, "confirmed": 4, ...},
  "total_confirmed": 13,
  "total_candidates": 13
}
```

### 3. 查看统计

```bash
curl http://localhost:9002/api/v1/binding/stats
```

### 4. 查看绑定详情

```bash
# 默认查看人才↔论文
curl "http://localhost:9002/api/v1/binding/detail?binding_type=talent_paper&limit=100"

# 查看人才↔专利
```


### 6. 清除绑定

```bash

# 清除绑定边 + 所有测试数据
```

## 前端页面

浏览器打开：

```
http://localhost:9002/binding
```

页面包含四个区域：

### 操作区
- **初始化数据** — 向 TRS Graph 插入测试节点、创建边类型和索引
- **执行全部绑定** — 运行三种绑定 Pipeline
- **人才↔论文 / 人才↔专利 / 机构↔机构** — 单独执行某类绑定
- **清除绑定 / 清除全部数据** — 清理数据重新来过

### 统计区
三张卡片显示：总绑定数、已确认（confirmed）、待确认（candidate）

### 绑定详情表
表格展示每条绑定的源实体→目标实体、置信度、匹配方法、规则分、LLM 分、状态

### 关系图
D3.js 力导向图，不同节点类型用不同颜色：
- 🔵 人才（talent）
- 🟢 论文（cn_paper）
- 🔴 专利（patent）
- 🟡 机构（cn_organization）

确认的绑定用实线，待确认的用虚线，边上标注置信度分数。节点可拖拽交互。

### 设计要点

测试数据故意包含以下挑战场景来验证绑定算法的鲁棒性：

- **缩写匹配**：「浙江大学」vs「浙大」— 规则召回应能通过编辑距离相似度匹配
- **跨语言**：「清华大学」vs「Tsinghua University」— 规则召回可能漏掉，LLM 应能识别
- **机构层级**：「清华大学」vs「清华大学计算机系」— 应识别为相关实体
- **无匹配项**：专利发明人「赵磊」不在人才库中 — 不应产生绑定
- **部门差异**：「清华大学」vs「清华大学计算机系」— 机构绑定应覆盖

## 绑定算法详解

### 规则召回

三种绑定类型使用不同的权重和阈值：

| 绑定类型 | 姓名权重 | 机构权重 | 字段权重 | 阈值 |
|---|---|---|---|---|
| 人才↔论文 | 0.6 | 0.3 | 0.1 | ≥ 0.5 |
| 人才↔专利 | 0.7 | 0.3 | — | ≥ 0.5 |
| 机构↔机构 | — | — | — | ≥ 0.6 |

相似度计算函数：
- **姓名**：精确匹配 → 1.0，否则用编辑距离相似度
- **机构**：取 Jaccard 相似度、编辑距离相似度、包含关系的最大值
- **字段**：字符级 Jaccard 相似度
- **机构加分**：同省份 +0.1，同城市 +0.1，同类型 +0.05

### LLM 精排

对每个规则召回的候选对，构造 Prompt 让 GLM 判断是否为同一实体：

```
你是一个实体对齐专家。请判断以下两个实体是否为同一个现实实体。

实体A（来自{源库}）：{字段摘要}
实体B（来自{目标库}）：{字段摘要}

请严格以JSON格式回答：
{"is_same": true/false, "confidence": 0.0-1.0, "reason": "判断理由"}
```

状态判定：
- `is_same=true` 且 `confidence ≥ 0.7` → **confirmed**
- `is_same=true` 且 `0.5 ≤ confidence < 0.7` → **candidate**
- 其余 → **rejected**（不创建边）

LLM 未配置时（`ZHIPUAI_API_KEY` 未设置），自动 fallback 到纯规则评分：`rule_score ≥ 0.7` 视为 confirmed。
