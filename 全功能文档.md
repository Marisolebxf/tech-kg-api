## 一、系统架构

### 整体设计

系统采用**策略模式 + 流水线架构**，所有功能通过注册中心统一管理，配置驱动执行。

```
API 请求
  ↓
策略注册中心（STRATEGIES）
  ↓
策略函数（Strategy Function）
  ↓
组件流水线（Pipeline）
  ├── Scorer（质量评分）
  ├── Mapper（数据转换）
  ├── Deduplicator（去重统计）
  └── Aggregator（维度聚合）
  ↓
结果返回
```

### 目录结构

```
dataset_factory/
├── config/                   # 配置文件
│   └── config.yaml           # 主配置文件
├── src/                      # 源代码
│   ├── component/            # 可复用组件
│   │   ├── aggregator/       # 聚合器（7个维度）
│   │   ├── deduplicator/     # 去重器（MD5/SimHash/MinHash）
│   │   ├── mapper/           # 数据转换器
│   │   ├── model_chat/       # 模型调用代理
│   │   ├── scorer/           # 质量评分器（15+个）
│   │   └── tokenizer/        # 分词器
│   ├── data/                 # 数据结构定义
│   ├── strategy/             # 策略实现（核心业务）
│   │   ├── data_eval/        # 数据集评测
│   │   ├── gov_doc/          # 政务文档处理
│   │   ├── hot_news_rewrite/ # 热点新闻改写
│   │   ├── knowledge_graph/  # 知识图谱QA生成
│   │   ├── nl2sql/           # NL2SQL生成
│   │   ├── safety_sft/       # 安全SFT增强
│   │   ├── scoring/          # 文本评分
│   │   ├── text_enhence/     # 通用文本增强
│   │   └── text_to_qa/       # 文本转问答
│   └── utils/                # 工具模块
└── test/                     # 测试用例
```

### 核心设计模式

| 模式 | 应用场景 |
|------|--------|
| 策略模式 | 注册中心动态加载和调用策略 |
| 代理模式 | ModelProxy 统一管理模型调用、并发控制、QPS 限制 |
| 装饰器模式 | `@STRATEGIES.mark()` 注册策略 |
| 工厂模式 | 根据策略编码创建对应的处理逻辑 |
| 流水线模式 | Scorer → Mapper → Deduplicator → Aggregator 串联执行 |

## 二、数据结构

### Sample（样本）

`src/data/sample.py`

样本是系统中的基础数据单元，支持预训练文本和问答对两种形式。

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | str | 样本唯一编码 |
| `content` | str | 预训练文本内容 |
| `instruction` | str | 问答对的指令（可选） |
| `question` | str | 问答对的问题 |
| `answer` | str | 问答对的答案 |
| `conversation` | List[Conversation] | 多轮对话（shareGPT格式） |
| `extra` | dict | 扩展字段（如 review_score、file_name 等） |
| `statistics` | dict | 各评分器的打分结果，key 为 scorer_name |
| `temp` | dict | 临时存储（接口返回前自动清空） |

**主要方法：**
- `is_pretrain_text()`：判断是否为预训练文本（有 content，无 question/answer）
- `is_single_chat()`：判断是否为单轮对话（有 question 和 answer）
- `get_text()`：获取文本内容（自动识别样本类型）
- `get_tokens()`：获取已缓存的分词结果

## 三、数据增强策略

### 3.1 通用文本增强（`general_text_cpt_enhance`）

`src/strategy/text_enhence/general_enhance.py`

在不改变主旨的前提下，使用 LLM 对文本进行增强改写。

**核心 Prompt（ENHANCE_PROMPT）设计：**
- 任务目标：剔除冗余、修复错误、提升可读性
- 约束：禁止输出推理/引导语等额外内容
- 引用三原则：无害性、有帮助性、诚实性

**可选角色扮演（ROLE_PLAYING_REQ）：**
当 `rolePlaying` 参数传入角色名时（如"交警"、"市民"、"政务人员"），自动注入风格化改写要求，模仿该角色的说话方式和语气。

**参数：**
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `enhanceModelCode` | deepseek-r1 | 用于增强的模型编码 |
| `rolePlaying` | 空 | 角色扮演风格（如"交警"、"市民"），传"无"或空则不启用 |

### 3.6 热点新闻改写（`hot_news_rewrite`）

`src/strategy/hot_news_rewrite/news_rewrite.py`

保持事实准确性的前提下，以多种风格重写热点新闻，用于构建新闻领域大模型训练数据。

**两阶段流水线：**

| 阶段 | Prompt | 默认模型 |
|------|--------|---------|
| 改写（NEWS_REWRITE_TEMPLATE） | 按风格重写新闻 | deepseek-r1 |
| 质检（QUALITY_CHECK_NEWS_PROMPT） | 4维度评分验证 | ernie-4.5-turbo-32k |

**支持改写风格（`rewriteStyle`）：**
- `official`：官方媒体风格，语言正式权威
- `popular`：通俗易懂风格，口语化接地气
- `analysis`：深度分析风格，专业严谨有深度
- `neutral`（默认）：客观中立风格，平实陈述事实

**质检维度（各 0-10 分）：**
- 事实准确性（Fact Accuracy）：核心事实是否正确，篡改则 0 分
- 内容完整性（Content Completeness）：是否保留全部关键信息
- 改写质量（Rewrite Quality）：是否有创新、非简单同义词替换
- 可读性（Readability）：语言是否清晰自然

**参数：**
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `rewriteModelCode` | deepseek-r1 | 改写模型 |
| `checkModelCode` | ernie-4.5-turbo-32k | 质检模型 |
| `rewriteStyle` | neutral | 改写风格 |
| `minTotalScore` | 7.0 | 质检通过最低分，低于此分重试 |
| `maxRetries` | 3 | 最大重试次数 |

**输出格式：**
```json
{"title": "改写后的标题", "content": "改写后的正文"}
```

## 四、数据生成策略

### 4.1 文本转问答对（`text_to_qa`）

`src/strategy/text_to_qa/text_to_qa.py`

从原始文本段落生成专业深入的 SFT 问答对，两阶段验证确保质量。

**两阶段流水线：**

| 阶段 | Prompt | 默认模型 |
|------|--------|---------|
| 生成（GEN_PROMPT） | 生成 1 个高质量 QA 对 | deepseek-r1 |
| 校验（VAL_PROMPT） | 4维度打分（10分制） | deepseek-r1 |

**生成 Prompt 核心约束：**
1. **文件名强制显式规则**：文本中出现法律法规名称时，问题必须完整写明全称，禁止"本办法"、"该规定"等代称
2. **答案质量红线**：
   - 禁止仅重复问题（"作出了具体要求"等空洞表述）
   - 禁止数字模糊表述（如"4个要求"，必须列出4条）
   - 禁止无实质定性描述（如"全面的规定"）
   - 合格答案需 ≥ 50-100 字具体描述
3. **Chain-of-Thought 引导**：15 步逐步推理任务流程（理解文本 → 分析前后文 → 生成问答 → 自检）
4. **上下文感知**：支持传入前置/后置文本（各最多 1000 字），短文本（<500字）自动通过 retriever 检索邻近段落

**校验维度（共 10 分）：**
- 问题专业性（3分）：是否使用领域术语，是否考察关键逻辑
- 答案准确性（3分）：是否严格基于原文，无外部信息
- 答案充实度（3分）：信息密度是否足够，无空洞表述
- 问答对应关系（1分）：答案是否直接回答问题

**参数：**
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `generateModelCode` | deepseek-r1 | 生成模型 |
| `validateModelCode` | deepseek-r1 | 校验模型 |
| `minScore` | 8 | 校验通过最低分（0-10） |
| `rolePlaying` | 空 | 角色扮演风格 |
| `fileId` | 空 | 文件 ID（用于检索上下文） |

### 4.2 NL2SQL 生成（`nl2sql`）

`src/strategy/nl2sql/nl2sql.py`

基于少量示例学习，生成新的问题-SQL 对，用于 NL2SQL 模型训练。

**两阶段流水线：**

| 阶段 | 功能 | 默认模型 |
|------|------|---------|
| 生成 | 逆向推导 schema，生成新问题和 SQL | deepseek-r1 |
| 验证 | 4维度质量检查 | ernie-4.5-turbo-32k |

**生成逻辑：**
1. 输入：少量示例（schema + 问题 + SQL 答案）
2. LLM 学习示例间的映射关系，逆向推导出不同但相似的数据库 schema
3. 基于新 schema 生成对应的自然语言问题和 SQL 答案
4. 确保生成内容与示例有显著差异，避免示例污染

**SQL 规范约束：**
- 正确使用日期函数、字段映射、条件过滤、聚合函数
- 保证 SQL 可执行性和语法正确性

**验证维度：**
- 准确性：SQL 是否正确映射问题需求
- 逻辑性：WHERE/JOIN/聚合逻辑是否正确
- 可执行性：SQL 语法检查
- 完整性：是否包含所有必要字段和条件

**输出格式：**
```json
{
  "prompt": "建表语句及约束",
  "query_generated": "自然语言问题",
  "response_generated": "对应 SQL 答案"
}
```

## 六、数据集评测

### 6.1 功能概述

数据集评测模块提供对数据集质量的全面评估，支持样本级和数据集级两个层次。

**接口地址：** `POST /api/v1/dataset/eval/process`

**评测模式：**
- `batch`：全量机评，支持分批提交，动态汇总数据集总分
- `single`：单条机评，快速评估单条样本
- 人工审核：用户手动修改样本总分，审核分数会覆盖机评分数

### 6.2 七个评测维度

#### 规范性（Normativity）

`src/component/aggregator/sample_normativity_aggregator.py`

**评分逻辑：**
- 问答对：问题和答案同时有效 → 1分，否则 0分
- 预训练文本：基于文本长度线性插值
  - 长度 < `char_length_thresh1`（默认10）：0 分
  - 长度 >= `char_length_thresh2`（默认128）：1 分
  - 两者之间：`(length - thresh1) / (thresh2 - thresh1)`

#### 多样性（Diversity）

`src/component/aggregator/sample_diversity_aggregator.py`

**评分逻辑（两个维度 L2 范数归一化）：**

- **词汇多样性（score1）**：
  - 文本长度 < sample_size（42）时：`score1 = TTR × 0.8`
    - TTR（Type-Token Ratio）= 不同词数 / 总词数
  - 文本长度 ≥ 42 时：`score1 = HDD`
    - HDD（Hypergeometric Distribution Diversity）= 超几何分布采样多样性，sample_size=42 是 vocd-D 论文中的最优经验值
- **文本不重复度（score2）**：自相关分析（auto_corr），通过 FFT 检测重复模式
  - 无重复峰值 → score2 ≈ 1
  - 存在强重复模式 → score2 接近 0

- **最终分数**：`(score1² + score2²)^0.5 / √2`（L2范数归一化到[0,1]）

#### 时效性（Timeliness）

`src/component/aggregator/sample_timeliness_aggregator.py`

**评分逻辑：**
- 调用外部 API 查询文档的法律状态
- `CURRENT`/`现行（1）` → 1 分
- 已废止/失效/未知 → 0 分

适用于法律法规类文档，通过文档的现行状态判断数据时效性。

#### 安全性（Safety）

`src/component/aggregator/sample_safety_aggregator.py`

**评分逻辑：**
- 调用 Qwen Guard 模型检测风险类别
- 按风险类别加权扣分

**风险类别权重：**
| 风险类别 | 权重 |
|---------|------|
| Violent（暴力） | 1.0 |
| Jailbreak（越狱） | 1.0 |
| Suicide & Self-Harm（自伤） | 1.0 |
| Non-violent Illegal（违法） | 0.9 |
| Sexual Content（色情） | 0.8 |
| Unethical Acts（不道德） | 0.7 |
| PII（个人隐私） | 0.5 |
| Politically Sensitive（政治敏感） | 0.4 |
| Copyright（版权） | 0.4 |

#### 干净性（Cleanliness）

`src/component/aggregator/sample_cleanliness_aggregator.py`

**评分逻辑（加权求和）：**
- **连贯度（score1，权重0.7）**：基于 PPL（文本困惑度）
  - PPL > `max_ppl_thresh`（默认1000）→ 0 分
  - 否则：`(max_ppl_thresh - PPL) / max_ppl_thresh`
  - 原理：PPL 越高 → 语言越不通顺 → 越可能是脏数据
- **特殊字符占比（score2，权重0.3）**：
  - 占比 > 0.1 → 0 分
  - 否则：`1 - ratio`

- **最终分数**：`score1 × 0.7 + score2 × 0.3`

#### 完整性（Integrity）

`src/component/aggregator/sample_integrity_aggregator.py`

**评分逻辑：**
- 使用 LLM 判断问答对/文本的完整性
- 评估是否存在截断、残缺、前后文不连贯等问题
- 返回 0-1 之间的分数

#### 准确性（Accuracy）

`src/component/aggregator/sample_accuracy_aggregator.py`

**评分逻辑：**
- 使用 LLM（`integrity_accuracy_scorer`）评估样本准确性
- 对问答对：检查答案是否准确回答问题
- 对预训练文本：检查事实陈述是否准确
- 评分模型：ernie-4.5-turbo-32k（默认）

### 6.3 评测流水线

```
样本输入
  ↓
[Scorer阶段] text_length_scorer → special_char_scorer → tokens_and_ppl_mapper
             → vocab_diversity_scorer → auto_corr_scorer → qwen_guard_scorer
             → integrity_accuracy_scorer → timeliness_scorer → ...
  ↓
[Aggregator阶段] 规范性 → 多样性 → 时效性 → 安全性 → 干净性 → 完整性 → 准确性
  ↓
[维度分数] 每个样本 statistics['dimensions'] 中存储7个维度分数
  ↓
[数据集级聚合] 计算数据集总分
  formula: dataset_score = sample_score_avg × 0.9 + dataset_diversity × 0.1
  ↓
[去重统计] MD5/SimHash/MinHash 三路去重，结果作为多样性补充指标
  ↓
[优化建议生成] 对低分维度采样低分样本，调用 LLM 生成改进建议
  ↓
返回评测报告
```

### 6.4 人工审核机制

- 通过 `extra.review_score`（0-10分）传入审核分数
- 通过 `extra.revoke_review = true` 撤销审核，恢复机评分数
- 审核分数会动态更新数据集总分
- 以生成/修改时间更靠后的结果为准

### 6.5 优化建议生成

`src/strategy/data_eval/eval_refine.py`

| Prompt 类型 | 触发条件 | 说明 |
|------------|---------|------|
| `DIMENSION_REFINE_PROMPT_TEMPLATE` | 维度分数 < 阈值（默认0.7） | 针对各维度低分原因生成改进建议 |
| `REVIEW_REFINE_PROMPT_TEMPLATE` | 审核样本分数 < 阈值（默认6.0） | 针对低分审核样本生成建议 |
| `FINAL_REFINE_PROMPT_TEMPLATE` | 汇总所有建议 | 生成数据集整体优化建议 |

### 6.6 并发安全设计

- **全局锁**：保护 `DatasetManager` 字典的创建/删除操作
- **Dataset 内部 RLock**：保护所有样本分数修改操作，支持多数据集并发评测
- **内存管理**：定时清理超时的 Dataset 管理器（默认超时 2 小时，可配置）
- **新轮次检测**：自动检测全量机评新轮次，清理旧数据避免混淆

## 十、去重器（Deduplicator）

去重器在**数据集级别**运行，统计重复率，结果用于多样性评分。

> 注意：去重器只做统计分析，不会自动删除重复样本。

`src/component/deduplicator/`

### 10.1 MD5 精确去重（`md5_deduplicator`）

- **原理**：对文本计算 MD5 哈希，哈希值相同即视为完全重复
- **适用场景**：爬虫重复抓取、同一文档多次入库等完全一致的重复
- **参数**：无（精确匹配，无阈值）
- **分组算法**：哈希值映射 + 并查集（Union-Find）

### 10.2 SimHash 近似去重（`simhash_deduplicator`）

- **原理**：对文本特征加权哈希生成 64 位指纹，用汉明距离衡量相似度
- **适用场景**：轻微改写、局部修改的近似重复
- **算法**：`simhash.find_all()` + 分块倒排索引 + BFS 连通分量

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `distance` | 3 | 汉明距离阈值（0=精确，3-5=长文本，5-10=宽松）|
| `blocks` | 4 | 分块数，必须 > distance |

### 10.3 MinHash 近似去重（`minhash_deduplicator`）

- **原理**：将文本表示为 n-gram 集合，MinHash 签名估算 Jaccard 相似度，LSH 快速查找候选对
- **适用场景**：段落增删、句子重排等集合相似的近似重复
- **签名存储**：base64 编码字符串，存入 `sample.statistics['minhash']`

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `threshold` | 0.7 | Jaccard 相似度阈值 |
| `num_perm` | 128 | 哈希置换次数（签名维度），越大越精确 |

### 三种去重方式对比

| 维度 | MD5 | SimHash | MinHash |
|------|-----|---------|---------|
| 去重类型 | 精确 | 近似 | 近似 |
| 相似度度量 | 完全相同 | 汉明距离 | Jaccard 相似度 |
| 签名大小 | 32位字符串 | 64位整数 | 128维向量 |
| 对词序 | 敏感 | 不敏感 | 不敏感 |
| 对局部修改 | 不容忍 | 容忍 | 容忍 |
| 计算开销 | 最低 | 中 | 中 |
| 重复组合并 | 并查集 | BFS连通分量 | 并查集 |

## 十一、映射器（Mapper）

映射器对样本进行数据转换，结果存入 `sample.temp` 或直接修改样本字段。

`src/component/mapper/`

### tokens_mapper（`tokens_mapper`）

对文本进行分词，支持多种分词模式。

**支持的分词模式：**
- `jieba`：中文分词
- `remote`：调用远端分词服务（推荐生产环境使用）
- `whitespace`：空格分词（英文）

**输出：** `sample.temp['tokens']`

### tokens_and_ppl_mapper（`tokens_and_ppl_mapper`）

分词同时计算困惑度（PPL），是干净性维度的核心依赖组件。

**按样本类型分别调用远端服务：**

- 预训练文本：发送 `content` → 返回 `content_tokens` + `content_perplexity`
- 问答对：
  - 发送 `answer` → 返回 `answer_tokens` + `answer_perplexity`
  - 发送 `instruction + question + answer`（拼接）→ 返回 `qa_tokens` + `qa_perplexity`

**IFD（指令跟随难度）计算：**
```python
# IFD = 问答对困惑度 / 答案单独困惑度
ifd_value = qa_perplexity / answer_perplexity
# IFD 越小 → 给定问题后答案困惑度下降越多 → 指令跟随越好
```

结果存入：`sample.statistics['instruct_quality']['ifd']`

**输出：** `sample.temp['tokens_and_ppl']`

### special_char_mapper（`special_char_mapper`）

清理文本中的特殊字符，直接修改 `sample.content/question/answer`。

## 十二、分词器（Tokenizer）

`src/component/tokenizer/`

### 远端分词器（remote_tokenizer）

通过 HTTP 调用远端分词服务，获取 tokens 和 PPL。

**两种调用模式：**
- **模式一（推荐）**：通过 `tokenizer_code` 使用 `TOKENIZER_PROXY`，支持并发控制和 QPS 限制
- **模式二**：直接通过 `url` 调用，适用于临时测试

**默认分词服务：** `qwen3guard-0.6b-ppl`（同时支持分词和 PPL 计算）

**接口返回格式：**

```json
{"tokens": ["词1", "词2", ...], "natures": ["n", "v", ...], "perplexity": 12.34}
```

**重试机制：** 默认最多重试 3 次，超时时间 30 秒
